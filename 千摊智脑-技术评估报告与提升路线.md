## 2. 必须立即修的 5 个真问题

### 🔴 P0-A 鉴权未覆盖全部写接口（越权黑洞）

| 位置 | 问题 |
|---|---|
| `routers/purchase.py:30`（修复前） | `merchant_id = uuid.UUID(body["merchant_id"])` —— 从**请求体**读身份 |
| `routers/purchase.py` `get_today_list` / `update_purchase_item` / `cancel_purchase_item` | 从 **query 参数**读 `merchant_id`，且**无 token 依赖** |
| `routers/purchase.py` `confirm_purchase` / `cancel_purchase_list` | **完全没有任何鉴权依赖** |

**风险**：任意客户端传入他人 `merchant_id`，即可读/改/确认/取消其采购单 → 跨商户数据泄漏 + 经营数据被篡改。这就是工作记忆里那条"P0 鉴权缺位"的实锤。

**已修（本次）**：`purchase.py` 全部端点已改为 `Depends(get_current_merchant)`，`confirm`/`cancel` 补了归属校验（`plist.merchant_id != merchant.id → 404`）。`py_compile` + 导入验证已通过。

**同类残留（本次未动，必须按同一模式修）**：
- `routers/voice.py:52` `merchant_id: uuid.UUID = Form(...)`、`:141` `uuid.UUID(body["merchant_id"])`
- `routers/vision.py:66` `merchant_id: uuid.UUID = Form(...)`

改法（统一模板）：函数签名加 `merchant: Merchant = Depends(get_current_merchant)`，用 `merchant.id` 替换所有 `merchant_id`，**删掉** Form/body 里的 `merchant_id`。`get_current_merchant` 在 `auth_allow_fallback=True` 时兼容旧前端（仍读 query），生产置 `False` 即强制——所以这是**向后兼容的安全升级**。

---

### 🔴 P0-B SKU 体系孤儿化（账本没接上"真实商品"）

**现象**：`catalog.py` 建了 `ProductSKU / ProductAlias / ProductSpecification / Unit / UnitConversion / Supplier`，文档白纸黑字写"SKU 取代 category 成为账本主键"。但——

- `InventoryRecord.product_id` → `product_categories.id`（Integer）
- `PurchaseItem.product_id` → `product_categories.id`（Integer）
- `BatchLifecycle.product_id` → `product_categories.id`（Integer）

**后果（很严重）**：
1. 语音"西红柿"→别名→SKU 解析完成后，**没有任何账本列能存 SKU**，解析结果落不了地。
2. 别名 / 规格 / 单位换算这些刚写好的能力，在真实记账链路上是**死的**。
3. AI 建议里的"西红柿"和"番茄"在账本里是两个 `product_categories`，成本和库存被割裂，建议永远算不准。

**修复路线（分阶段，别硬切，避免数据迁移事故）**：
1. 三张表加 `sku_id`（UUID FK `product_skus.id`），与现有 `product_id` **并存**。
2. 写一次性 backfill：按 `product_categories.name` 匹配/新建 `ProductSKU`，回填 `sku_id`。
3. `voice_parser` 解析后写 `sku_id`；库存聚合优先按 `sku_id` group（SKU 名即 canonical name）。
4. `product_id` 仅作兼容/去规范化，新功能一律用 `sku_id`。

> 这是 AI 建议准确、单位换算生效、一批一码溯源的**前提**，必须在第一阶段深入前完成。否则后面所有"真实账本"功能都建在沙子上。

---

### 🟠 P1-C 金额用 `float`（违反你们自己定的红线 #7）

**现象**：`InventoryRecord` 的 `quantity/unit_cost/unit_price/total_amount`、`PurchaseItem` 各金额列都是 `sa.Numeric` 但 `Mapped` 类型是 `float`；`ProductSKU.default_sale_price` 同理。

**后果**：Python 端聚合/对账用 `float`，二进制浮点误差会让"日结差异"出现几分钱偏差——摊主对账时对系统失去信任，比报错更糟。

**修复**：本次新表（`accounts` / `price_history`）已用 `from decimal import Decimal` + `Mapped[Decimal]` 做**样板**。存量列需一次类型修正迁移：把 `Mapped[float]` 改为 `Mapped[Decimal]`，序列化层 `float(x)` 仅用于出参（DB 内部保持 `Decimal`）。低风险，建议跟着 P0-B 的迁移一起做。

### 附录：AI 建议一键执行的 service 模板（#7 闭环示范）

```python
# services/advice_executor.py —— 让"立即改价"真改价，而不是只显示已采纳
from decimal import Decimal
from sqlalchemy import select
from app.models.catalog import ProductSKU, PriceHistory
from app.models.audit import AuditLog

async def execute_price_change(
    db, merchant_id, sku_id, new_price: Decimal, reason: str, changed_by: str
) -> PriceHistory:
    sku = await db.get(ProductSKU, sku_id)
    if sku is None or sku.merchant_id != merchant_id:
        raise ValueError("SKU 不存在或无权操作")
    old = sku.default_sale_price or Decimal("0")
    sku.default_sale_price = new_price
    ph = PriceHistory(
        merchant_id=merchant_id, sku_id=sku_id,
        old_price=old, new_price=new_price, reason=reason,
        source="ai", changed_by=changed_by,
    )
    db.add(ph)
    db.add(AuditLog(merchant_id=merchant_id, action="price_change",
                     target_table="product_skus", target_id=str(sku_id),
                     after_data={"old": str(old), "new": str(new_price), "reason": reason},
                     operator=changed_by))
    await db.commit()
    return ph   # 返回落库记录，前端据此更新价签状态
```

要点：**改价 = 改 SKU 售价 + 写 PriceHistory + 写审计**，三件事在一个事务里完成。这样 AI 建议系统才能事后复盘"我那条建议被执行了没、效果如何"。
