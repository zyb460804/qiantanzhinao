> 身份 `merchant_id` 由客户端任意指定，全站无鉴权。这不是某个文件的 bug，而是横跨 11 个路由、25+ 处接口签名共同踩中的**架构级坑**。它直接让你们所有的"多租户隔离""离线同步多商户""差分隐私"都建筑在沙滩上。

一句话定级：**安全地基缺位 + 规范未强制 + 路由层过胖** —— 这三点正好是团队技术能力从"能跑"跨到"可维护/可上线"的瓶颈。下面的方案就是冲着这三点去的。

## 1. 代码质量把控 — 分级发现（Code Review）

严重度： **P0** 安全/数据泄露 · **P1** 正确性/一致性 · **P2** 可维护性 · **P3** 工程化

### 🔴 P0-1 鉴权缺位：merchant_id 由客户端任意指定（系统级）

- **证据**：`app/routers/*.py` 至少 25 处把 `merchant_id` 当作普通参数（Query / Body / Form），部分代表：
  - `advice.py:30`、`behavior.py:20,59`、`inventory.py:23,70,110,541`
  - `purchase.py:129,189,239`、`reports.py:33,75,251,423,488`
  - `twin.py:22,58,116,201,235`、`vision.py:66`(Form)、`voice.py:52,177,192`
- **风险**：任何客户端传别人的 UUID 即可读/写其全部经营数据。**多租户隔离形同虚设**。`main.py:42-52` 的 12 个 `include_router` 中**无任何 auth 路由**，`get_current_*` 全文无匹配。
- **最恶劣的一处**：`simulate_scenario`（`advice.py:222`）在缺失时 `merchant_id = uuid.UUID(body.get("merchant_id", str(uuid.uuid4())))` —— **缺失就随机造一个商户身份**，等于默认以"幽灵商户"执行。
- **教学点（给全员背下来的军规）**：**身份永远来自可信凭证（token / session），绝不来自请求体/查询参数。** 这是 Web 安全 ABC，也是本项目头号技术债。修它，比加 10 个功能都重要。

### 🔴 P0-2 CORS 配置自相矛盾

- **证据**：`main.py:33-39` `allow_origins=["*"]` **且** `allow_credentials=True`。
- **风险**：带凭证的跨域绝不应使用通配符。该组合在浏览器中被拒绝、或退化为"任意源可带凭证访问"，等于对跨站读取/ CSRF 敞开。
- **应写**：明确列出受信前端域名（生产环境）；开发期可放宽但必须带注释说明。

### 🟠 P1-1 响应契约未落地：`response_model=dict` 绕过自有 Schema

- **证据**：`advice.py:28` 的 `get_daily_advice` 用 `response_model=dict`，但团队已写好 `schemas/advice.py:23 DailyAdviceResponse` / `DailyAdviceItem` 却没接上。
- **风险**：失去响应校验、OpenAPI 文档失真、前端无类型提示；更隐蔽的是**"写了规范却绕开规范"**，会向团队传递错误信号——规范是可选的。
- **应写**：`response_model=DailyAdviceResponse`，让 FastAPI 自动校验 + 序列化 + 生成文档。

### 🟠 P1-2 接口入参回归裸 dict

- **证据**：`simulate_scenario`（`advice.py:216-237`）用 `body: dict = Body(...)`，而同文件 `simulate_what_if_endpoint` 用的是 `WhatIfRequest`。同一文件两种风格。
- **风险**：无字段校验、无类型、无文档；`scenario` 缺字段时运行时才崩。
- **应写**：补 `ScenarioBatchRequest(BaseModel)`，出参 `response_model=list[SimulationOutput]`。

### 🟡 P2-1 路由层过胖（Fat Router）

- **证据**：`get_daily_advice`（`advice.py:28-171`）约 80 行，内联了 3 类 DB 查询 + 循环 + 业务编排。
- **风险**：难单测（强依赖 DB）、复用困难、PR 审查成本高。
- **应写**：路由只做"取依赖 → 调 service → 返回"。把编排抽到 `app/services/advisor.py:build_daily_advice(db, merchant_id)`。你们已有 `advisor` service，方向对了，只是没贯彻到每个接口。

### 🟡 P2-2 N+1 查询

- **证据**：`get_daily_advice` 在 `for product in products:` 里对每个商品各发 3 条查询（库存 / 7 日 / 30 日）。
- **风险**：商品数增长时查询数线性膨胀；SQLite 下尚可，PostgreSQL 下成瓶颈。
- **应写**：用 `group_by(InventoryRecord.product_id)` 一次性聚合，或预加载。

### 🟡 P2-3 模块内重复 import

- **证据**：`advice.py:80` 与 `:190` 在函数体内 `from datetime import timedelta`，应提到文件头。
- **风险**：小，但暴露"随手写"习惯；CI 的 `ruff` 本应拦下（E402）。

### ⚪ P3-1 异常被静默吞掉

- **证据**：`advice.py:114` `except Exception as fe:` 兜底预测失败 → 返回 `None`，仅 warning 日志。
- **风险**：预测失败对业务是"静默降级"，线上难定位。
- **应写**：结构化日志（含 merchant / product / error type）+ 指标计数器；必要时显式返回"预测不可用"状态，而非假装成功。

## 2. 参考实现（Senior 给出"正确长什么样"）

> 完整可运行参考见附件 **`千摊智脑-auth参考实现.py`**。
> **这是参考代码，请团队 code review 后再决定是否合入，不要直接覆盖现有文件。**

它演示三件事：

1. **`get_current_merchant` 依赖**：身份只来自 Bearer token，绝不来自请求体。
2. **`simulate_scenario` 改造**：用 Pydantic Schema + 从 token 取 merchant_id + `response_model` 落地。
3. **`get_daily_advice` 瘦路由拆分**：编排逻辑进 service，路由只调一行。

核心片段（节选）：

```python
bearer_scheme = HTTPBearer(auto_error=False)

async def get_current_merchant(
    creds: HTTPAuthorizationCredentials | None = Depends(bearer_scheme),
    db: AsyncSession = Depends(get_db),
) -> Merchant:
    """身份只来自 token，绝不接受客户端传来的 merchant_id。"""
    if creds is None or not creds.credentials:
        raise HTTPException(401, "Missing authentication token",
                            headers={"WWW-Authenticate": "Bearer"})
    merchant_id = _decode_merchant_id_from_token(creds.credentials)  # 真实 JWT 校验
    merchant = await db.get(Merchant, merchant_id)
    if merchant is None:
        raise HTTPException(403, "Merchant not found")
    return merchant

@router.post("/simulate/scenario", response_model=list[SimulationOutput])
async def simulate_scenario(
    body: ScenarioBatchRequest,                         # 类型安全入参
    merchant: Merchant = Depends(get_current_merchant), # 身份只来自 token
    db: AsyncSession = Depends(get_db),
):
    return [SimulationOutput(**simulate_what_if(**s.model_dump())) for s in body.simulations]
```

## 3. 团队技术提升路线（分阶段 · 每阶段有验收点）

> 设计原则：**先堵安全洞（P0），再立契约纪律（P1），然后瘦身与数据层（P2），最后工程化门禁（P3）。** 每阶段都用现有代码当教材，边改边学。

### 阶段 0 — 安全地基（最高优先级，约 1 周）
- **目标**：消灭 P0-1 / P0-2。落地微信登录（`wx.login` → code2Session 换 openid → 绑 `Merchant` → 签发 JWT）；所有写接口从 token 取 `merchant_id`；收紧 CORS。
- **学习材料**：FastAPI Security（HTTPBearer / OAuth2）、PyJWT、小程序登录流程。
- **验收**：① 任意接口无法用他人 `merchant_id` 读到数据；② 自动化测试覆盖"越权访问被 403"；③ CORS 生产配置无 `*` + credentials。

### 阶段 1 — 契约一致性（约 2 周）
- **目标**：`response_model` 全量落地；消灭裸 `dict` 入参；确立"返回信封 `{code,data}` vs HTTP 状态码"的团队统一约定（建议：业务错误用信封，系统/鉴权错误用 HTTP 状态码）。
- **学习材料**：FastAPI `response_model`、Pydantic v2、OpenAPI。
- **验收**：`grep` 无 `response_model=dict`、无 `body: dict`；CI `ruff` 强制。

### 阶段 2 — 路由瘦身与数据层纪律（约 2–3 周）
- **目标**：路由 < 30 行；service 层承载业务逻辑；N+1 清零；Alembic 取代 `create_all` 成为唯一迁移手段（`main.py` 仅在测试/SQLite dev 用 `create_all`）。
- **学习材料**：分层架构、SQLAlchemy 聚合查询、Alembic 迁移工作流。
- **验收**：新增接口必须经 service；任何模型变更走 Alembic 迁移脚本。

### 阶段 3 — 质量门禁工程化（持续）
- **目标**：pre-commit（`ruff` + `mypy`）+ CI 流水线；code review checklist 执行；典型问题归档 `docs/bug-archive.md`。
- **学习材料**：pre-commit、CI（GitHub Actions / 你们司内 CI）、`mypy` 渐进类型。
- **验收**：本地提交即跑 lint/type；PR 必须有 review；新增 P0 缺陷 = 0。
