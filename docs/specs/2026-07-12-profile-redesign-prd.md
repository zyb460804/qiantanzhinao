# 「我的」页面重设计 PRD

> **版本**: v1.0 | **日期**: 2026-07-12 | **作者**: 产品通 | **状态**: 待评审

---

## 一、问题陈述

当前「我的」页面本质上是一个「设置页 + 工具快捷入口」，缺失小微商户最核心的诉求：

1. **没有经营数据**：摊主想快速看「今天赚了多少」，只能跳转经营台或报告页，至少多一步
2. **设备状态不可见**：树莓派 / 摄像头是否在线？离线数据有没有上传？摊主完全不知道
3. **没有帮助渠道**：遇到问题没有快速求助入口
4. **工具入口不分层**：12 个入口平铺，高频和低频没有区分，新用户迷失

**不解决的成本**：用户留存差（「我的」是最高频进入的 TabBar 页之一，但当前几乎无信息增量）、设备问题无法及时发现导致数据断档、缺乏帮助渠道导致流失。

---

## 二、目标

| # | 目标 | 衡量方式 |
|---|------|---------|
| 1 | 摊主打开「我的」3 秒内获取今日经营概况 | 页面首屏加载到营收数字渲染 < 1.5s |
| 2 | 高频操作（语音记账/收银/查看经营）减少 1 次页面跳转 | 快捷操作区点击率 > 30% |
| 3 | 设备异常可被摊主自主发现 | 设备异常状态下，设备面板点击率 > 50% |
| 4 | 提供完整的帮助反馈闭环 | 反馈提交 → 7 日内响应率 100% |

---

## 三、Non-Goals（本版本不做）

| # | 不做的事 | 原因 |
|---|---------|------|
| 1 | 个性化推荐 / 智能提醒卡片 | 依赖 AI 规则引擎重构，属于下一阶段 |
| 2 | 消息通知中心 | 推送体系尚未建立，先做页面内信息透出 |
| 3 | 多摊位/连锁店管理 | 当前目标用户是单摊位小微商户 |
| 4 | 经营成就/徽章系统 | 有趣但非刚需，优先保证核心功能完整 |
| 5 | 深色模式下单独适配 | 沿用现有 `ui-dark` 类，不新增独立深色设计 |

## 五、用户故事

### 经营快照

> 作为摊主，我打开「我的」页面时，希望第一眼就看到今日营收和交易情况，以便用最短路径确认今天的经营状况。

> 作为摊主，当我今天还没有任何交易记录时，我希望看到清晰的引导，以便我立即去记账。

### 快捷操作

> 作为摊主，我希望最常用的功能（语音记账、收银开单）在一个显眼的位置，以便我不需要在工具网格里找。

> 作为摊主，当我采购清单里有待购商品时，我希望在「我的」页面能被提醒，以便我不会忘记补货。

### 设备与同步

> 作为摊主，我希望知道我的树莓派和摄像头是否正常运行，以便在设备断连时及时发现。

> 作为摊主，我希望知道离线记账数据是否成功上传，以便确认数据不会丢失。

### 帮助与反馈

> 作为摊主，当我遇到使用问题时，我希望能在「我的」页面找到帮助或直接反馈，以便不用退出小程序去其他渠道求助。

### 模块 ①：经营快照

#### 布局

```
┌─────────────────────────────────────────────┐
│  🏪 张姐蔬菜摊                                │
│  ─────────────────────────────────────────── │
│                                              │
│   ¥ 1,286.50         ↑ +12% 较昨日           │
│   今日营收                                    │
│                                              │
│  ┌──────────┐ ┌──────────┐ ┌──────────┐     │
│  │  23 笔    │ │ ¥55.93   │ │  ▸ 持平  │     │
│  │ 交易笔数  │ │ 客单价    │ │ 经营趋势  │     │
│  └──────────┘ └──────────┘ └──────────┘     │
│                                              │
│  📊 本周累计 ¥6,832 →                        │
└─────────────────────────────────────────────┘
```

#### 设计要点

| 要素 | 决策 | 理由 |
|------|------|------|
| 营收大字 | 36rpx 粗体，颜色 `var(--ink)` | 用户第一眼锚定到金额 |
| 日环比 | `↑/↓ +XX% 较昨日`，涨绿跌红 | 比绝对值更重要的上下文 |
| 三指标横排 | 笔数 / 客单价 / 趋势 | 紧凑不占高度，一眼扫完 |
| 本周累计 | 底部弱提示，末尾 `→` | 引导跳转经营报告 |
| 空状态 | 「今天还没有记录，去记一笔吧 →」 | 点击跳转语音记账页 |
| 加载态 | 骨架屏：营收位置矩形占位闪烁 | 避免白屏闪跳 |
| 错误态 | 「数据加载失败，下拉重试」 | 不影响页面其余部分 |

#### 数据来源

| 字段 | API | 路径 |
|------|-----|------|
| 今日营收 | `GET /api/v1/twin/dashboard` | `.today_revenue` |
| 昨日营收 | `GET /api/v1/reports/daily` | `.yesterday_revenue` |
| 日环比 | 前端计算 | `(today - yesterday) / yesterday * 100` |
| 交易笔数 | `GET /api/v1/reports/daily` | `.sale_qty` |
| 客单价 | 前端计算 | `today_revenue / sale_qty` |
| 本周累计 | `GET /api/v1/reports/daily` | `.week_total_revenue`（如无，用 trends 7 天聚合） |

> **注意**：当前后端一次调用需要两个接口（`/twin/dashboard` + `/reports/daily`）。建议前端并发请求，或后端新增聚合接口 `GET /api/v1/profile/summary` 合并返回。

#### 日环比颜色规则

| 条件 | 颜色 | 说明 |
|------|------|------|
| 涨 > 0% | `var(--success)` 绿色 | 今天比昨天好 |
| 跌 < 0% | `var(--tomato)` 红橙 | 今天比昨天差 |
| 持平 = 0% | `var(--muted)` 灰色 | 无变化 |

#### 经营趋势规则

| 条件 | 显示 |
|------|------|
| 连续 3 天上涨 | `↗ 向好` 绿色 |
| 连续 3 天下跌 | `↘ 走弱` 红橙 |
| 波动 | `↕ 波动` 灰色 |
| 数据不足（< 3 天） | `— 待观察` 灰色 |

- [ ] 今日交易笔数
- [ ] 今日客单价
- [ ] 空数据状态（无交易时引导去记账）
- [ ] 骨架屏加载态
- [ ] 网络错误降级（显示错误提示，不影响其他模块）

### 模块 ②：快捷操作

#### 布局

```
┌─────────────────────────────────────────────┐
│  快捷操作                                    │
│  ─────────────────────────────────────────── │
│  ┌──────────┐ ┌──────────┐ ┌──────────┐     │
│  │  🎤       │ │  📊       │ │  🧾       │     │
│  │ 语音记账  │ │ 经营镜像  │ │ 收银开单  │     │
│  │ 今天还没记│ │ 今日全貌  │ │ 快速开单  │     │
│  └──────────┘ └──────────┘ └──────────┘     │
│  ┌──────────────────────────────┐           │
│  │  📋  采购清单         待补 3 件│  ← 动态角标│
│  │      检查需要进什么货          │           │
│  └──────────────────────────────┘           │
└─────────────────────────────────────────────┘
```

#### 设计要点

| 要素 | 决策 | 理由 |
|------|------|------|
| 排位 | 语音记账 > 经营镜像 > 收银开单 > 采购清单 | 按摊主日常操作频率排序 |
| 布局 | 前三 3 列等宽 + 采购清单通栏 | 采购清单不同频，但有待办时需要眼睛 |
| 副标题 | 每卡片一行小字（12-14rpx） | 降低认知成本 |
| 语音记账动态文案 | 今日已记 →「再记一笔」；未记 →「今天还没记」 | 制造温和的驱动力 |
| 采购清单角标 | 有未购商品时红色圆点+N；为空时不显示 | 不做虚假紧迫 |

#### 交互
- 点击卡片 → `wx.navigateTo()` 跳转对应页面
- 采购清单角标数字来自 `app.globalData.pendingPurchaseCount` 或 `wx.getStorageSync('purchaseList')`

#### 设备状态判断规则

| 最后心跳 | 状态 | 颜色 |
|----------|------|------|
| < 5 分钟 | 在线 | 🟢 绿色 |
| 5-30 分钟 | 不稳定 | 🟡 琥珀色 |
| > 30 分钟 或 null | 离线 | 🔴 红色 |

#### 离线同步状态

| 队列状态 | 显示 |
|----------|------|
| 待上传 0 条 | 「已同步」绿色 |
| 待上传 1-5 条 | 「N 条待上传」琥珀色 |
| 待上传 > 5 条 或上次同步失败 | 「N 条待上传」红色 + 点击重试 |

队列数量从客户端 `localStorage` 的离线同步队列读取（`offline-queue` 或 `offline-sync.js` 中的队列）。

#### 交互
- 设备行点击 → `wx.navigateTo('/pages/devices/devices')`
- 离线行点击 → 有失败时触发手动同步
- 「查看全部设备 →」→ 跳转设备管理页

## 九、风险与合规

| 风险 | 影响 | 缓解措施 |
|------|------|---------|
| 经营快照两个接口并发失败率 | 首屏数据缺失 | 任一成功即展示部分数据；双失败降级为空状态 |
| 离线队列 localStorage 读取延迟 | 同步状态显示不准 | 只读不写，操作在 `offline-sync.js` 中完成 |
| 微信客服会话组件兼容性 | 低版本微信无此功能 | 降级为拨打电话或展示二维码 |
| **隐私合规** | 经营数据涉及商户敏感信息 | 所有 API 调用走 JWT 认证；不在客户端持久化存储营收明细 |
| 设备心跳数据被恶意利用 | 暴露商户营业规律 | 设备状态仅当前商户可见，后台校验 `merchant_id` |
