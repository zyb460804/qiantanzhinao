# 千摊智脑管理后台 V2 升级蓝图

> 版本：V2.0  
> 基线日期：2026-07-13  
> 适用范围：`qiantan-brain/backend/admin-web`、`backend/app/routers/admin`、`backend/app/routers/tenant` 及 SaaS 多租户能力  
> 定位：从“可用的 SaaS 管理 CRUD”升级为“平台运营、客户成功、商业化、AI 治理一体化控制台”

---

## 1. 结论先行

当前后台已经不是“从零建设”，而是一个**后端能力较完整、前端已具备 MVP、但产品化与工程化不足**的 SaaS 管理后台。

下一阶段不建议继续横向堆普通 CRUD 页面，建议围绕以下四条主线升级：

1. **平台运营化**：让管理员快速发现异常租户、临期订阅、超配额、欠费、设备离线和经营风险。
2. **租户工作台化**：把租户详情从普通资料页升级为客户成功工作台，集中呈现订阅、用量、商户、设备、风险和操作历史。
3. **AI 可治理化**：把现有建议、AIAction、语音、视觉能力纳入后台，形成模型用量、建议质量、动作执行、安全边界和反馈闭环。
4. **工程生产化**：补齐 TypeScript、权限守卫、查询缓存、自动测试、按路由拆包、错误监控、安全会话和可观测性。

建议产品名称统一为：

- 平台端：**千摊智脑 · 平台运营中心**
- 租户端：**千摊智脑 · 组织管理台**
- 摊主端：保留微信小程序，不与平台后台混合

### 2.3 现阶段主要差距

| 等级 | 问题 | 影响 |
|---|---|---|
| P0 | 仅“已登录”控制，无完整前端 RBAC；后端虽有 `super_admin/ops_admin` 字段，但未形成细粒度权限依赖 | 运营管理员可能访问计费、套餐等高权限操作 |
| P0 | JWT 存储于 localStorage | 一旦发生 XSS，令牌可被直接窃取 |
| P0 | 租户隔离主要依赖业务代码和上下文，尚需系统性验证所有业务表与查询 | 存在跨租户数据泄漏风险 |
| P1 | 没有管理员账号、角色、权限、审计日志前端页面 | 无法形成完整治理闭环 |
| P1 | 当前租户详情仅资料编辑，未形成租户 360° 工作台 | 客户成功和故障排查效率低 |
| P1 | 没有 AI 运营与治理后台 | 无法评估建议质量、执行率、模型成本和风险 |
| P1 | 前端 JavaScript、无统一数据层、无自动化测试/检查脚本 | 迭代后回归风险快速上升 |
| P1 | 页面以独立 Card/Table 为主，缺少统一页面模板、状态规范和设计令牌 | 视觉与交互一致性不足 |
| P2 | 暗黑模式不持久化，主题仅局部设置 | 体验不稳定 |
| P2 | 缺少错误边界、全局异常页、403/404、请求重试与取消 | 弱网和接口异常体验差 |
| P2 | 订阅、发票后端能力大于前端能力 | 创建、调整、生成账单等流程不完整 |
| P2 | 无系统健康、任务队列、设备告警、通知中心 | 平台运维仍需查看日志或数据库 |

## 3. V2 产品目标与指标

### 3.1 北极星目标

让平台运营人员在一个后台内完成：

> 发现问题 → 定位租户 → 判断影响 → 执行操作 → 留下审计 → 验证结果

### 3.2 建议核心指标

#### 平台经营

- 活跃租户数
- 试用转付费率
- 月度经常性收入 MRR
- 续费率、流失率
- ARPU
- 逾期账单金额

#### 产品使用

- DAU/WAU 租户数
- API 调用量
- 活跃商户数
- 关键功能渗透率：POS、库存、采购、AI 建议、视觉、语音
- 配额达到 80%/100% 的租户数

#### AI 质量

- 建议生成数
- 建议采纳率
- AIAction 执行成功率
- 被拒绝/撤销率
- 单租户 AI 成本
- 语音解析纠错率
- 视觉识别人工修正率

#### 稳定性

- API 成功率
- P95 响应时长
- 异常租户数
- 离线设备数
- 同步积压数
- 关键任务失败数

## 5. 核心页面升级设计

### 5.1 平台总览

#### 页面目标

不是展示“总数”，而是帮助管理员判断今天是否需要处理问题。

#### 推荐布局

1. 顶部全局筛选：时间范围、套餐、租户状态、区域。
2. 第一行 KPI：活跃租户、MRR、试用转化、逾期金额、API 成功率。
3. 第二行趋势：租户增长、收入趋势、调用量趋势。
4. 第三行经营结构：套餐分布、功能使用渗透率、租户健康度分布。
5. 右侧/底部待办：试用即将到期、用量超 80%、账单逾期、设备离线、同步失败。
6. 最近动态：创建租户、套餐变更、人工改账、管理员操作。

#### 交互要求

- KPI 必须显示环比/同比和数据口径。
- 点击卡片直接进入已带筛选条件的列表页。
- 图表必须有空状态、异常状态、下载数据入口。
- 不使用纯装饰图表；每个图表都应回答一个运营问题。

### 5.2 租户列表

推荐字段：

- 租户名称/Slug
- 状态
- 套餐
- 商户数/上限
- API 用量/上限
- 近 7 日活跃度
- 健康度
- 试用或订阅到期日
- 欠费状态
- 创建时间
- 负责人/客户成功经理（后续）

推荐批量操作：

- 批量导出
- 批量发送通知
- 批量分配负责人
- 批量调整标签

禁止直接在列表提供无二次确认的“停用/删除”。

### 5.3 租户 360° 详情工作台

将当前“详情 + 编辑”升级为以下标签页：

1. **概览**：健康度、风险、关键指标、最近活动、待办。
2. **组织资料**：基本资料、联系人、标签、平台备注。
3. **商户与成员**：商户、摊位、管理员、角色。
4. **订阅与账单**：套餐、周期、历史变更、账单、人工调整。
5. **用量与配额**：当前用量、趋势、超限历史、配额变更。
6. **设备与同步**：设备状态、最后在线、边缘节点、同步积压。
7. **AI 使用**：建议、动作、成功率、成本、人工反馈。
8. **风险与审计**：食品安全、异常行为、管理员操作日志。

右侧固定操作区：

- 修改资料
- 调整套餐
- 延长试用
- 暂停/恢复租户
- 生成账单
- 重置管理员登录
- 模拟进入租户（必须强审计、限时、明显水印）

### 5.4 套餐与权益

升级重点：

- 套餐价格和权益必须版本化，避免修改套餐后影响历史订阅解释。
- `features` JSON 不应直接让运营编辑原始 JSON，改为功能开关表单。
- 套餐需要“草稿、已发布、已下线”状态。
- 删除改为停用；已有订阅引用时禁止物理删除。
- 展示受影响租户数量和迁移预览。

### 5.5 订阅与账单

建立清晰状态机：

```text
trialing → active → past_due → suspended → canceled/expired
```

要求：

- 所有状态变更显示原因、操作人、生效时间。
- 升降级先展示计费差额和生效策略。
- 人工标记已支付必须填写支付方式、流水号、备注。
- 账单作废、退款、冲正不能覆盖原记录，应新增反向记录。
- 金额字段统一 Decimal，前端统一分为最小货币单位或字符串传输。

### 5.6 用量监控

当前页面按租户选择后查看概览，V2 应增加：

- 全平台用量排行
- 超 80%、超 100% 租户列表
- 指标维度筛选
- 日/周/月粒度
- 异常突增检测
- 配额调整历史
- 用量明细追溯

人工记账属于高风险操作，必须：

- 限定权限
- 填写原因
- 二次确认
- 写入审计日志
- 支持反向冲正，不允许直接删除历史记录

### 5.7 AI 运营中心

这是“千摊智脑”区别于普通 SaaS 后台的核心模块。

#### AI 总览

- 活跃 AI 租户
- 建议生成量
- 建议采纳率
- AIAction 执行成功率
- 模型调用量与成本
- 高风险动作数
- 失败原因 Top N

#### 建议与动作

- 查看建议输入摘要、规则/模型版本、输出、置信度、执行动作和结果。
- 对改价、采购、清货、库存调整等动作建立风险等级。
- 高风险动作要求人工确认，禁止默认自动执行。
- 支持按租户、商品、动作类型、状态、时间筛选。
- 支持运营标记“有效/无效/有害/需优化”。

#### 提示词版本管理

每个提示词版本记录：

- prompt_key
- 版本号
- 场景
- 系统提示词
- 输入变量 Schema
- 输出 JSON Schema
- 模型与参数
- 风险等级
- 灰度租户
- 生效时间
- 评测结果
- 创建人/审批人

发布流程：草稿 → 测试 → 灰度 → 全量 → 回滚。

## 6. 视觉设计规范

### 6.1 品牌方向

关键词：**可信、专业、经营感、克制的智能化、农贸行业温度**。

避免：

- 大面积紫色渐变
- 过度科技蓝光效
- 每个卡片都使用彩色背景
- 把 AI 页面做成聊天机器人集合

推荐色彩：

```css
:root {
  --brand-600: #167A5A;
  --brand-500: #1F8F68;
  --brand-100: #E8F5EF;

  --info-600: #2563EB;
  --success-600: #16803C;
  --warning-600: #D97706;
  --danger-600: #DC2626;

  --text-primary: #17201C;
  --text-secondary: #52605A;
  --text-tertiary: #849089;
  --border: #E5EAE7;
  --bg-page: #F5F7F6;
  --bg-card: #FFFFFF;

  --radius-sm: 6px;
  --radius-md: 10px;
  --radius-lg: 14px;
  --shadow-card: 0 1px 2px rgba(17, 36, 27, .05), 0 6px 18px rgba(17, 36, 27, .04);
}
```

### 6.2 字体与密度

- 正文：14px / 22px
- 辅助文字：12px / 20px
- 页面标题：24px / 32px，600
- 区块标题：16px / 24px，600
- KPI 数字：28–32px，600
- 表格默认中等密度，行高 52–56px
- 金额和指标建议使用等宽数字特性 `font-variant-numeric: tabular-nums`

### 6.3 间距

采用 4px 基础网格：4、8、12、16、20、24、32、40。

- 页面内边距：24px；窄屏 16px
- 卡片间距：16px
- 区块间距：24px
- 表单字段纵向间距：20px
- 筛选区与表格间距：16px

### 6.4 状态色

- 正常/已支付/在线：绿色
- 处理中/试用/待确认：蓝色
- 即将到期/接近配额：橙色
- 失败/欠费/停用/高风险：红色
- 已取消/无效/离线：灰色

颜色不能作为唯一信息载体，必须配合文字或图标。

## 7. 页面与交互规范

### 7.1 四类统一页面模板

#### 列表页

页面标题 → 简介 → 主操作 → 筛选区 → 批量操作 → 表格 → 分页。

#### 详情页

对象标题 → 状态 → 关键操作 → 核心摘要 → Tabs → 时间线/关联数据。

#### 仪表盘

全局筛选 → KPI → 趋势 → 结构 → 排行/明细 → 口径说明。

#### 表单页

基础信息 → 业务配置 → 风险提示 → 预览 → 提交。

### 7.2 加载、空、错、无权限

每个数据区块必须具备：

- Skeleton 首屏加载
- 局部 Spin 刷新
- 空状态：解释为什么为空，并给出下一步操作
- 错误状态：错误摘要、Request ID、重试按钮
- 403：说明缺少哪个权限，不只是“无权限”
- 404：提供回到列表或搜索入口

### 7.3 危险操作

以下操作必须使用确认弹窗，并要求输入原因：

- 停用租户
- 取消订阅
- 手工改账
- 标记已支付
- 重置密钥
- 删除/停用套餐
- 模拟进入租户
- 执行高风险 AI 动作

确认文案必须说明：对象、影响、生效时间、是否可撤销。

### 7.4 表格规范

- 默认固定关键首列和操作末列。
- 超过 8 列支持列设置。
- 筛选条件同步到 URL，刷新后保留。
- 导出必须继承当前筛选条件。
- 日期统一 `YYYY-MM-DD HH:mm`，并明确时区。
- 金额右对齐，数量右对齐，文本左对齐。
- 操作超过 3 个时收进“更多”。

### 7.5 文案规范

- 按钮用“动词 + 对象”：新建租户、生成账单、调整套餐。
- 避免笼统的“确定”“提交”，除非上下文绝对明确。
- 错误信息结构：发生了什么 + 可能原因 + 下一步。
- 成功提示说明结果，例如“已暂停租户，现有登录将在 5 分钟内失效”。

## 8. 权限与审计规范

### 8.1 权限模型

采用 RBAC + 数据范围：

```text
用户 → 角色 → 权限点 → 数据范围
```

建议平台角色：

| 角色 | 核心权限 |
|---|---|
| super_admin | 全部权限、管理员与权限管理 |
| ops_admin | 租户运营、用量、设备、AI 运营；不可改价和改账 |
| billing_admin | 套餐、订阅、账单、收款；不可管理系统权限 |
| support_admin | 只读租户数据、工单、有限的协助操作 |
| auditor | 只读审计、安全、账单与变更记录 |

建议权限点命名：

```text
tenant.read
tenant.create
tenant.update
tenant.suspend
subscription.read
subscription.change
invoice.read
invoice.create
invoice.mark_paid
usage.read
usage.adjust
ai_action.read
ai_action.approve
admin.manage
audit.read
```

### 8.2 双端校验

- 前端权限只用于隐藏/禁用交互。
- 后端权限是最终安全边界。
- 每个写操作路由必须显式声明权限点，不使用“只要是管理员即可”。
- 数据导出、模拟登录和批量操作必须单独授权。

### 8.3 审计要求

审计日志至少记录：

- 操作人、角色
- 动作
- 资源类型和资源 ID
- 变更前/变更后
- 原因
- IP、User-Agent
- Request ID
- 时间
- 结果：成功/失败

审计日志禁止被普通管理员删除或修改。

## 9. 前端工程升级

### 9.1 推荐目录

```text
src/
├─ app/                  # 路由、Provider、权限、错误边界
├─ api/                  # 生成的 API client、请求拦截器
├─ assets/
├─ components/           # 通用组件
├─ features/             # 按业务域组织
│  ├─ dashboard/
│  ├─ tenants/
│  ├─ billing/
│  ├─ usage/
│  ├─ ai-ops/
│  ├─ devices/
│  └─ security/
├─ layouts/
├─ pages/                # 路由入口，保持薄
├─ permissions/
├─ theme/
├─ types/
└─ utils/
```

### 9.2 技术升级建议

- JavaScript → TypeScript，开启 `strict`。
- 使用 TanStack Query 管理服务端状态、缓存、失效和重试。
- 表单统一使用 Ant Design Form + Zod/类型 Schema（二选一保持一致）。
- 路由页面 `React.lazy` 动态加载。
- 将 Recharts 仅放入需要图表的异步 chunk。
- 统一 API 错误结构，不在页面内重复 try/catch 文案。
- 增加 ErrorBoundary、403、404、500 页面。
- 增加 ESLint、Prettier、Vitest、React Testing Library、Playwright。
- 主题和暗色模式持久化，优先跟随系统设置。

### 9.3 性能门槛

- 首屏主包 gzip 目标：小于 250 kB。
- 单路由异步 chunk gzip：建议小于 150 kB。
- LCP：桌面正常网络小于 2.5s。
- 表格 1000+ 条数据使用服务端分页；超大明细考虑虚拟列表。
- 搜索输入 300ms 防抖，并取消过期请求。

## 10. 后端与 API 升级

### 10.1 响应协议

统一响应：

```json
{
  "code": 0,
  "message": "ok",
  "data": {},
  "request_id": "req_xxx"
}
```

统一分页：

```json
{
  "items": [],
  "page": 1,
  "page_size": 20,
  "total": 100,
  "has_next": true
}
```

统一错误：

```json
{
  "code": "SUBSCRIPTION_INVALID_TRANSITION",
  "message": "当前订阅状态不能激活",
  "details": {},
  "request_id": "req_xxx"
}
```

### 10.2 必补后端能力

1. 平台管理员 CRUD、角色与权限点。
2. 后端 `require_admin_permission()` 依赖。
3. 租户状态和订阅状态的显式状态机。
4. 套餐版本、权益快照、订阅变更历史。
5. 用量冲正与变更历史。
6. 审计变更前/后快照和失败记录。
7. 通知与告警中心。
8. 系统健康、任务状态、同步积压 API。
9. AI 调用记录、提示词版本、评测与反馈模型。
10. 幂等键：创建账单、标记支付、订阅变更等写操作。

### 10.3 多租户安全验证

- 对所有包含租户数据的模型建立租户字段清单。
- 对所有读写 API 增加跨租户负向测试。
- 禁止前端传入 tenant_id 后被后端无条件信任。
- 平台跨租户查询必须使用独立、显式的平台权限依赖。
- 导出和异步任务同样必须携带租户上下文。

## 11. 登录与安全升级

### 11.1 推荐会话方案

优先方案：

- Access Token 放 HttpOnly、Secure、SameSite Cookie。
- CSRF Token 使用双提交或服务端 Token。
- Refresh Token 轮换，支持单设备/全部设备退出。
- 管理后台强制较短会话时长和空闲超时。

如短期仍使用 localStorage：

- 严格 CSP
- 禁止内联脚本
- 依赖漏洞扫描
- Token 短有效期
- 刷新令牌不存 localStorage
- 对高风险操作要求二次验证

### 11.2 高风险能力

- 超级管理员启用 MFA。
- 模拟进入租户显示持续水印和倒计时。
- 下载敏感数据可增加水印、审批或二次验证。
- 登录失败限流、异常地点提醒、密码策略和账号锁定。

## 12. 实施路线图

### Phase 0：稳住基线（2–3 天）

- 补充 admin-web README 与环境变量说明。
- 增加 ESLint/Prettier/build 检查。
- 路由懒加载与手工分包。
- 增加 403/404/500、ErrorBoundary。
- 修复主题持久化和统一错误提示。

验收：构建无大包警告或主包 gzip 小于 300 kB；所有路由可独立加载。

### Phase 1：产品化升级（5–8 天）

- 新版平台总览。
- 租户列表升级。
- 租户 360° 工作台。
- 订阅与账单流程补全。
- 审计日志页面。
- 统一页面模板和设计令牌。

验收：运营人员无需访问数据库即可完成租户开通、调整套餐、生成账单、查用量、看审计。

### Phase 2：权限与安全（4–6 天）

- 平台管理员管理。
- 权限点与角色矩阵。
- 后端细粒度权限依赖。
- 高风险操作二次确认与原因。
- 会话安全升级。
- 跨租户安全测试。

验收：`ops_admin` 无法修改套餐价格、标记付款或管理管理员；所有拒绝均有审计和 Request ID。

### Phase 3：AI 与设备运营（6–10 天）

- AI 总览。
- 建议与动作治理。
- 提示词版本管理。
- 设备中心、离线告警、同步积压。
- AI/设备异常告警。

验收：可追溯一次 AI 建议从输入、版本、输出到动作执行结果的完整链路。

### Phase 4：商业化增强（持续迭代）

- 自动续费、支付回调、对账。
- 套餐版本与价格实验。
- 客户成功健康度。
- 通知中心、工单、SLA。
- 数据仓库/BI 指标层。

## 14. Definition of Done

每个后台功能完成必须同时满足：

- 有权限点和后端校验。
- 有加载、空、错、403 状态。
- 写操作有成功反馈；危险操作有影响说明和确认。
- 有审计日志。
- 有 TypeScript 类型。
- 有至少一个正常测试和一个异常/无权限测试。
- 支持 Request ID 定位问题。
- 列表筛选可通过 URL 复现。
- 金额、时间、状态文案符合统一规范。
- 通过 lint、test、build。
- 不产生新的明显大包或敏感信息泄漏。

## 15. 项目决策建议

1. **先做单体模块化，不拆微服务。** 当前规模下 FastAPI 模块化足够，优先保证租户隔离和可观测性。
2. **平台端与租户端分产品域。** 可共用组件与 API 包，但路由、权限和导航独立。
3. **AI 必须可追溯、可评测、可回滚。** 不把 Prompt 写死在散落业务代码中。
4. **账务记录不可覆盖。** 用追加、冲正和状态历史替代直接修改历史事实。
5. **所有“自动执行”默认从人工确认开始。** 通过数据证明安全后再逐级放开自动化。
