## 统一走 Alembic 收口（去除 create_all 启动建表依赖）

延续 P0-B 轮已生成的全量基线 `5242218be814`，本轮把"统一走 Alembic"从过渡方案推进到终点。

### 背景（为何之前是过渡）
- P0-B 轮发现仓库实际只有一条全量基线 `5242218be814`（autogenerate 从空库对比模型生成，含 sku_id/账户表等全部 26 表），但 main.py 当时用过渡：`init_db()`(create_all 建表) + `run_alembic_migrations()`(stamp head 让旧增量 001-004 变 no-op、只跑 005)。create_all 仍是启动建表源。
- 本轮会话初 versions/ 里又出现 001-005（增量迁移，001 假设原始表已存在 → `alembic upgrade head` 直接报 "no such table: inventory_records"，证实链从未真正跑通，app 一直靠 create_all 掩盖）。

### 做了什么
- 备份并删除坏掉的旧增量 001-005（`_legacy_backup` 已清），保留全量基线 `5242218be814` 作为唯一迁移。
- 漂移检测（临时脚本，已删）：对比 `alembic upgrade head` 产物 vs `Base.metadata.create_all` 产物，26 表完全一致、零漂移。
- `migrations/env.py`：`run_migrations_online` 支持从 `context.config.attributes["engine"]` 注入 engine，复用 app 自身 engine 建表到同一库（兼容 in-memory/文件 SQLite）；无注入时仍按 settings.database_url 新建（CLI 行为不变）。
- `app/database.py`：`init_db()` 改为优先 `await asyncio.to_thread(_run_alembic_upgrade, engine)`（Alembic 内部 asyncio.run，须丢独立线程规避 running-loop 冲突）；dev(debug=True) 失败回退 create_all，prod(debug=False) fail-fast。去掉旧的 `run_alembic_migrations`/`_build_alembic_config`。
- `app/config.py`：加 `run_migrations_on_startup`（默认 True）。
- `app/main.py`：lifespan 简化为只 `await init_db()`。
- venv 补装 `alembic==1.13.0`（requirements.txt 有但 venv 漏装，导致迁移此前从未真正跑过）。
- 临时验证脚本与 `_legacy_backup` 已删；旧 `qiantan_dev.db` 删不掉（被占用，不影响测试；真实启动 dev 会告警+回退 create_all，建议手动删或换新库）。

### 验证
- 端到端：`init_db()` 经 Alembic 注入 engine 从空库建出 26 业务表 + `alembic_version`(5242218be814)，sku_id/外键/is_voided 抽查通过。
- pytest **105 passed**（测试不触发 lifespan，conftest in-memory + create_all 独立建表，零影响）。

### 结论（可复用铁律）
"统一走 Alembic"技术债 ✅ 已收口：Alembic 成为唯一启动建表权威，从空库即可完整建表，不再依赖 create_all。未来加列/表：先 `alembic revision --autogenerate` 挂到 `5242218be814` head 之下，勿再造 00x 孤儿链（之前 004/005 即此坑）。
- **async 上下文跑 Alembic 的坑**：`alembic.command.upgrade` 内部 `asyncio.run`，在 app lifespan 的 running loop 里直接调用会报 "cannot run loop while another loop is running"；解法 `await asyncio.to_thread(command.upgrade, cfg, "head")`，把同步 Alembic 调用丢到无 running loop 的线程。
- **env.py 注入 engine 的坑**：app 启动时把自身 engine 注入 `cfg.attributes["engine"]`，env.py `run_migrations_online` 优先用注入 engine，保证 Alembic 建表到与 app 同一库（尤其 in-memory SQLite 测试场景）。
