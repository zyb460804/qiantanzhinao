# 千摊智脑 · 新功能产品规格文档包（v0.1）

> **生成日期**：2026-07-11
> **作者角色**：产品通（Product Management Expert）
> **文档定位**：开发与设计团队可直接据此开工的 PRD 合集 + 需求补盲
> **范围决策**：依据《功能缺口与机会清单》（G1–G10）与商户隐私优先原则整理
> **核心取舍**：**G3 同行激励榜已砍**（与"私人商户数据私密"取向冲突）；隐私作为贯穿所有功能的横切非功能需求

---

## 0. 立项目的与范围决策

### 0.1 为什么是这批功能
- 当前已落地：语音记账、视觉识货、库存盘点、经营报告、采购清单、AI 参谋（Prophet）、经营镜像/数字孪生、经验云聚合、离线同步（PRD 已出）、消息通知（PRD 已出）。
- 仍"裸奔/纯 mock"且**阻塞业务闭环**的：多商户鉴权（G1）、收款台（G2）、经营日历（G4）。它们是用户能直接感知价值、且工程可行性高的部分，优先成 PRD。
- G8/G9/G10 复杂度或依赖度高，本文给规格要点，提示独立深化。

### 0.2 砍 G3 的正式说明
原 G3「同行激励榜」设计为 `GET /league/weekly` + `/league/quests`，即便内置差分隐私与匿名化，本质仍做**跨商户相对经营位置的横向对比**。这与本产品"私人商户数据私密"的核心取向冲突——任何排行榜都会间接暴露"谁强谁弱"的相对信号，摊主无动力公开。故**本期不做 G3**，相关社交激励诉求留待"纯个体目标达成"（如自我冲榜里程碑）形态另议。

### 0.3 隐私作为横切主题（见 §6）
所有功能默认遵循：数据最小化、传输加密、存储加密、跨商户隔离（G1 实现）、差分隐私（经验云已落地）、隐私同意、被遗忘权。

## 1. 遗漏需求发现（PM 主动补盲 · 以"商户数据私密"为透镜）

> 以下为从既有缺口清单**之外**挖出的需求，多数与"数据归属、控制权、安全、可用性"相关，是摊主真实但未被显性提出过的诉求。

| 编号 | 新需求 | 为什么摊主需要（Why） | 优先级 | 关联功能 |
|---|---|---|---|---|
| N1 | **数据所有权与可携/被遗忘权** | 当前无任何数据导出/删除接口。摊主换手机、退租、转行时，经营数据无法带走或彻底清除，违反《个人信息保护法》第 15/45 条（撤回同意、删除权）。 | P1 | G1（鉴权）、全局 |
| N2 | **首次启动隐私同意（Consent）** | 当前无隐私政策告知与授权环节。语音/图像/经营数据实时采集，需在使用前明确告知用途并获同意，否则合规风险高。 | P0 | G1 |
| N3 | **多角色/多设备（夫妻店权限）** | 摊点常由家人共同经营，但当前"一个 Merchant = 一个账号"。需店主/店员分级（店员不可删账、不可看利润明细），且数据仍归属同一商户。 | P2 | G1 |
| N4 | **风控与异常交易预警** | 误记、重复收款、异常大额无校验。需基于历史基线识别异常并提醒，保护商户资金与数据质量。 | P2 | G2、全局 |
| N5 | **适老化与无障碍** | 摊主年龄跨度大，部分视力/操作力弱。当前依赖小字与手动输入。需字号放大、高对比、语音播报增强、关键流程极简化。 | P1 | 全局 UI |
| N6 | **数据备份与灾难恢复** | 清缓存/卸载即丢（G2 现状即为教训）。需云端自动备份 + 一键恢复，防丢单。 | P1 | 离线同步、G2 |
| N7 | **操作审计对外可追溯** | 已有 `AuditLog` 模型（作废/修正留痕），但摊主无法自查"谁在何时改了什么"。需在前端开放只读审计视图，增强信任。 | P2 | 全局 |

**补盲结论**：N1/N2 是合规底线，应随 G1 一同落地；N5/N6 是"可用性 + 防丢"的体验刚需，排 Next；N3/N4/N7 为 Later 增强。

## 2. PRD-G1 · 多商户登录与鉴权（P0 · 安全地基）

### 2.1 功能概述与背景说明
- **背景**：`models/merchant.py` 已有 `Merchant` 模型，但 `routers/advice.py` 中 `merchant_id` 由客户端传入、缺失时随机生成；`main.py` 12 个 router 中**无任何 auth 路由**。当前是"信任客户端"的裸模型，任意客户端可伪造他人 `merchant_id` 读写其数据，多租户隔离形同虚设。
- **目的**：建立可信身份，使"多商户隔离"从查询侧过滤升级为身份侧强制，为离线同步、收款台、Web 后台等所有功能提供安全地基。
- **与现有 PRD 关系**：离线同步 PRD §4.1.2/§6.2 的"多商户隔离"实际依赖本功能才能闭环，**建议先于或同步于离线同步落地**。已有 `千摊智脑-auth参考实现.py` 可作实现蓝本。

### 2.2 目标用户群体及使用场景
- **目标用户**：全部千摊智脑摊主（微信小程序用户），含夫妻店多操作者（见 N3）。
- **场景**：
  - S1 首次启动 → 微信授权登录 → 自动建/绑 Merchant → 进入主页。
  - S2 日常启动 → 静默用本地 JWT 续期 → 无感使用。
  - S3 换设备/清缓存 → 重新微信登录 → 恢复同一商户数据。
  - S4（N2）首次启动 → 先弹隐私同意 → 同意后才采集语音/图像。

### 2.3 功能需求清单（按优先级）
| 优先级 | 需求 | 说明 |
|---|---|---|
| P0 | 微信登录链路 | `wx.login` → code → 后端换 openid → 绑定/创建 `Merchant` → 签发 JWT |
| P0 | `get_current_merchant` 依赖 | 所有**写接口**从 token 取 `merchant_id`，不再信任 body |
| P0 | 登录态校验中间件 | 未携带/失效 token 的受保护接口返回 401 |
| P0 | 隐私同意（N2） | 首次启动 Consent 页，同意后写标记，否则限制数据采集 |
| P1 | Token 刷新机制 | 短期 access + refresh，无感续期 |
| P1 | 登出 | 吊销 token，清除本地凭据 |
| P2 | 多角色（N3） | 店主/店员分级权限，数据归属同一商户 |

### 2.4 用户交互流程
```
启动 → 检查本地 JWT
  ├─ 有效 → 静默进入主页
  └─ 无效/无 → 引导微信授权
         → wx.login 取 code
         → POST /auth/login {code}
         → 后端换 openid → 建/绑 Merchant → 签发 JWT
         → 存 storage → 进入主页
任意写请求 → Header 带 Authorization: Bearer <JWT>
         → 中间件校验 → get_current_merchant 注入 merchant_id
```

### 2.5 非功能性需求
- **安全性**：JWT HS256/RS256 签名、access 有效期 ≤ 2h、refresh ≤ 7d、全程 HTTPS；openid 不与设备绑定泄露；token 存 `wx.setStorageSync` 并防截屏。
- **性能**：登录链路端到端 < 800ms（P95）。
- **兼容性**：微信基础库 ≥ 2.27（wx.login 能力）；iOS/Android 一致。
- **隐私**：openid/经营数据加密存储；Consent 记录可追溯。

### 2.6 验收标准与测试要点
- [ ] 任意写接口不带 token → 401。
- [ ] 伪造他人 `merchant_id` 于 body → 被 token 身份覆盖/拒绝，跨商户读不到。
- [ ] 同一微信用户重复登录 → 绑定同一 Merchant（幂等）。
- [ ] 隐私同意前 → 语音/图像采集被拦截。
- [ ] 并发 100 商户登录 → 无串号、无死锁。
- [ ] token 过期 → 自动刷新后请求成功。

### 2.7 发布计划与依赖项
- **阶段**：Now（P0）。先于离线同步上线。
- **依赖**：微信小程序 appid/secret 配置；`Merchant` 模型已存在。
- **风险**：微信登录偶发失败 → 需重试 + 友好降级提示；JWT 密钥轮换需平滑。

## 3. PRD-G2 · 收款台后端（P0 · 闭环收款）

### 3.1 功能概述与背景说明
- **背景**：`pages/pos/pos.js` 全程仅 `wx.setStorageSync` + `globalData`，**无任何 request 调用**；`main.py` 无 cashier 路由；语音播报仅为 Toast 文本模拟。收款金额不落库，卸载/清缓存即丢，且无法进入经营报告/AI 参谋计算。
- **目的**：让收款台成为真实业务入口，金额可靠落库、可进报表、可离线、有真实语音播报。
- **与现有 PRD 关系**：离线同步 PRD §7.2 D1 明确收款台后端为阻塞项，两者合并排期。

### 3.2 目标用户群体及使用场景
- **目标用户**：现场收银的摊主。
- **场景**：
  - S1 顾客付款 → 输入/选择金额与方式（微信/支付宝/现金）→ 收款落库 → 真实语音播报"收款 X 元"。
  - S2 网络中断 → 本地队列缓存 → 联网自动补传（复用离线同步 `client_id` 幂等）。
  - S3 日终 → 收款汇总自动进经营日报/参谋。

### 3.3 功能需求清单（按优先级）
| 优先级 | 需求 | 说明 |
|---|---|---|
| P0 | `POST /cashier/receipt` | 收金额/方式/时间，返回记录；`client_id` 幂等（复用离线 PRD 规范） |
| P0 | 真实语音播报 | 改用 `wx.createInnerAudioContext`（TTS 或预录金额音频），替代 Toast |
| P0 | 离线缓存+补传 | 断网先存本地队列，联网按 `client_id` 去重上传 |
| P1 | 进报表计算 | 收款记录纳入日报/周报/参谋 |
| P1 | 当日收款汇总接口 | `GET /cashier/today-summary` |
| P1 | 退款/撤销 | 原路或记账冲正，留审计 |

### 3.4 用户交互流程
```
收款台 → 输入金额 + 选支付方式 → 点"收款"
  → 生成 client_id（UUID）
  → 网络可用：POST /cashier/receipt {amount, method, client_id}
       → 落库 → 返回记录 → 播报"收款 X 元" → 更新首页金额
  → 网络不可用：写入本地队列 → 播报（本地）→ 联网后自动补传（去重）
```

### 3.5 非功能性需求
- **性能**：收银高峰单商户并发 ≥ 5 笔/秒无丢失；播报延迟 < 500ms。
- **安全性**：金额不可篡改（服务端校验 + 审计）；支付方式为枚举约束。
- **兼容性**：微信支付/支付宝/现金三种方式；iOS/Android 音频一致。
- **离线**：与离线同步 PRD 共用 `client_id` 幂等与队列规范。

### 3.6 验收标准与测试要点
- [ ] 收款成功落库，刷新/重进不丢。
- [ ] 同一 `client_id` 重复提交 → 仅一条记录（幂等）。
- [ ] 断网收款 → 联网后自动补传且无重复。
- [ ] 语音播报为真实音频（非 Toast）。
- [ ] 收款金额进入当日日报统计。
- [ ] 并发 50 笔收款 → 无丢失、无串单。

### 3.7 发布计划与依赖项
- **阶段**：Now（P0），与离线同步 D1 合并评审。
- **依赖**：G1 鉴权（写接口身份）；离线同步 PRD 的 `client_id` 规范。
- **风险**：音频播报在部分机型延迟；离线队列与在线状态判断需健壮。

## 4. PRD-G4 · 经营日历/时令后端（P1 · 环境感知）

### 4.1 功能概述与背景说明
- **背景**：`pages/calendar/calendar.js` 时令与策略为本地 mock（小暑固定文案）；`main.py` 无此路由。天气服务 `weather.py`（QWeather + 节假日）已就绪可直接复用。
- **目的**：让"时令备货/营销/时段"建议随真实节气与本地天气动态变化，兑现"环境感知"卖点。

### 4.2 目标用户群体及使用场景
- **目标用户**：按节气/天气调整备货与促销的摊主。
- **场景**：
  - S1 打开日历页 → 看到真实当前节气 + 当令品类。
  - S2 收到三行建议：备货 / 营销 / 最佳时段，随天气变化（如下雨推热饮、高温推冰品）。

### 4.3 功能需求清单（按优先级）
| 优先级 | 需求 | 说明 |
|---|---|---|
| P0 | `GET /env/solar-term` | 真实节气 + 当令品类（结合 `_get_holiday()` 与天气） |
| P0 | `GET /advice/seasonal` | 时令 + 本地天气 → 备货/营销/时段三行建议，喂给日历页 |
| P1 | 节假日/天气组合策略 | 节假日客流 + 天气叠加的精细化建议 |
| P1 | 数据来源标识 | 返回 `source: qweather/mock/cache`，前端展示来源 |

### 4.4 用户交互流程
```
日历页加载 → 并行调 /env/solar-term + /advice/seasonal（带位置/城市）
  → 渲染时令卡（节气 + 当令品类）
  → 渲染三行建议（备货/营销/时段）
  → 标注数据来源
外部天气 API 失效 → 降级返回 mock/缓存，UI 不崩、标"示例"
```

### 4.5 非功能性需求
- **性能**：日历页首屏接口 < 600ms（P95）。
- **兼容性**：天气服务限流时降级；无定位权限时回退城市默认。
- **可靠性**：外部依赖失败不影响核心记账功能。
- **隐私**：位置信息仅用于天气查询，不上报经营云。

### 4.6 验收标准与测试要点
- [ ] 返回真实当前节气（非写死）。
- [ ] 模拟下雨/高温 → 建议内容相应变化。
- [ ] 天气 API 失效 → 降级 mock，UI 正常且标注来源。
- [ ] 无定位权限 → 回退默认城市，不报错。
- [ ] 建议可点击跳转备货/采购页。

### 4.7 发布计划与依赖项
- **阶段**：Next（P1），与 G5 消息通知同批（共享"主动智能"主线）。
- **依赖**：`weather.py` 已就绪；无需新硬件。
- **风险**：天气 API 配额/限流；节气算法边界（跨日零点）。

## 6. 隐私合规横切要求（贯穿所有功能）

1. **数据最小化**：仅采集实现功能所必需的数据；位置信息仅用于天气查询，不进经营云。
2. **传输与存储加密**：全链路 HTTPS；经营数据、openid 加密存储。
3. **跨商户隔离**：由 G1 鉴权强制实现，查询侧 `merchant_id ==` 过滤作为二次防线。
4. **差分隐私**：经验云聚合已落地 `epsilon`/`Laplace`/分桶/查询预算，新聚合类功能复用。
5. **隐私同意（N2）**：首次启动明确告知并获授权，可撤回。
6. **被遗忘权（N1）**：提供数据导出与删除接口，满足个保法第 45 条。
7. **审计可追溯（N7）**：关键操作留 `AuditLog`，前端开放只读视图。
