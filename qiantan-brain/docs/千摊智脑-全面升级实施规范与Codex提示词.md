# 千摊智脑：全面升级功能清单、问题整改规范与 Codex 实施提示词

> 文档用途：统一前几轮项目检查、问题修复和真实业务升级需求，作为后续产品、开发、测试、部署和 Codex 自动实施的唯一工作基线。
>
> 项目定位：**从进货、验收、称重、销售、收款、库存、报损到日结，尽量靠说话和少量点击完成经营的生鲜摊位智能账本。**

---

## 1. 总体目标

项目不能继续停留在“AI经营驾驶舱 Demo”，应升级为可供摊主每天使用的经营系统。系统必须能回答：

1. 货从哪里来？
2. 实际进了多少货？
3. 钱付给了谁、是否还欠款？
4. 库存为什么变化？
5. 货卖给了谁？
6. 钱是否真正到账？
7. 哪些商品发生损耗，原因是什么？
8. 今天、这一周、这个月到底赚了多少？
9. AI建议是否真正执行、执行后是否有效？
10. 某个批次出现问题时，如何锁定、下架、召回和追溯？

### 1.1 产品原则

- 业务流水是真相，禁止只维护可随意覆盖的余额。
- 已确认业务记录禁止直接删除，必须作废、冲正或生成反向流水。
- 核心经营功能必须支持弱网和离线。
- AI不只“展示建议”，必须能触发真实业务动作并追踪效果。
- 操作复杂度必须适合菜市场摊主，不能照搬大型超市ERP。
- 自动识别允许不完美，但必须提供人工快速确认和纠错。
- 所有商户数据必须隔离，任何查询和写入都必须绑定当前登录商户。

## 4. 完整业务功能清单

## 4.1 商品、SKU、别名、规格和单位

必须支持：

- 标准商品 `Product`。
- 商品分类 `ProductCategory`。
- 商品SKU `ProductSKU`。
- 商品别名 `ProductAlias`。
- 商品规格 `ProductSpecification`。
- 单位 `Unit`。
- 单位换算 `UnitConversion`。
- 供应商商品映射 `SupplierProduct`。
- 价格历史 `PriceHistory`。
- 同一商品多个俗称，如番茄、西红柿、洋柿子。
- 精品、普通、次品、大果、小果等规格。
- 采购按筐、袋、件，销售按斤、公斤、盒、个。
- 固定换算和本次采购动态换算并存。
- 每个SKU独立采购成本、销售价、保质期和库存。

示例语音：

> 从老王那里进了三筐精品西红柿，一共142斤，花了310块，先欠着。

系统应解析并生成供应商、SKU、包装数、净重、总成本、单位成本、应付账款和入库批次。

## 4.2 供应商管理

必须支持：

- 供应商档案、联系方式、地址。
- 经营品类和供应商品。
- 历史报价。
- 最小起订量。
- 预计到货时间。
- 默认账期。
- 证照和凭证。
- 缺斤率、退货率、质量问题率、准时率。
- 关联批次损耗率。
- 供应商综合评分。
- 停用和黑名单。

## 4.3 采购完整闭环

状态机：

```text
草稿 → 待下单 → 已下单 → 部分到货 → 待验收 → 已验收 → 已入库 → 已完成
                                      ↘ 退货/补货/取消
```

必须支持：

- AI建议转采购单。
- 手工新建采购单。
- 选择供应商。
- 预计数量、价格和到货时间。
- 部分到货。
- 毛重、皮重、净重。
- 包装数量和动态单位换算。
- 采购单与实际到货差异。
- 缺斤少两。
- 质量验收。
- 到货、退货和补货照片。
- 合格证、检疫证、进货凭证。
- 现付、赊账、部分付款和预付款。
- 验收后自动批次入库。
- 自动生成供应商应付流水。
- 采购退货抵扣应付。
- 供应商对账单。

## 4.4 库存统一流水

建议统一模型：

- `StockLedger`
- `StockTransaction`
- `StockBalance`
- `WasteRecord`
- `StocktakeSession`
- `StocktakeDifference`

库存事件至少包括：

- 采购入库。
- 销售出库。
- 顾客退货。
- 退供应商。
- 腐损报废。
- 试吃赠送。
- 盘点增加或减少。
- 批次锁定和解锁。
- 不合格下架。
- 召回和销毁。
- 冲正。

库存状态必须区分：

- 账面库存。
- 可售库存。
- 锁定库存。
- 预占库存。
- 报损库存。
- 待同步库存。

规则：

- 已确认记录不能直接删除。
- 修改必须生成冲正或反向流水。
- 每次变化必须有来源类型和来源ID。
- 支持幂等。
- 支持FIFO批次扣减。
- 防止负库存。
- 并发销售时必须加锁或使用原子条件更新。

## 4.5 POS销售

必须支持：

- 商品快捷键。
- 搜索和扫码。
- 重量或数量录入。
- 智能秤重量带入。
- 单品改价。
- 整单折扣。
- 抹零。
- 挂单、取单。
- 多顾客连续称重。
- 现金、微信、支付宝、银行卡、赊账。
- 组合支付。
- 支付不足和找零。
- 小票。
- 离线下单。
- 重复请求幂等。
- 付款成功后扣库存。
- 库存不足整单失败。

## 4.6 退款、退货和订单冲正

必须支持：

- 整单退款。
- 部分商品退款。
- 部分金额退款。
- 退款但不退货。
- 退货重新入可售库存。
- 退货进入不可售库存。
- 原支付渠道退款。
- 现金补退。
- 退款审批。
- 退款原因。
- 退款幂等。

禁止删除原订单。退款必须产生：

- 退款单。
- 负向支付流水。
- 必要的正向库存流水。
- 审计日志。

## 4.7 支付与自动对账

数据模型建议：

- `PaymentTransaction`
- `PaymentChannel`
- `SettlementRecord`
- `ReconciliationTask`
- `ReconciliationDifference`

必须支持：

- 一单多支付流水。
- 支付回调幂等。
- 交易账单下载。
- 资金账单下载。
- 微信支付、退款和手续费对账。
- 系统有订单但无到账。
- 有到账但无订单。
- 金额不一致。
- 重复收款。
- 未匹配收款。
- 差异人工处理和审计。

## 4.8 客户应收和供应商应付

客户应收：

- 客户档案和联系方式。
- 信用额度。
- 默认账期。
- 赊账销售。
- 部分回款和全额回款。
- 逾期提醒。
- 停止赊账。
- 月度对账单。

供应商应付：

- 采购产生应付。
- 部分付款。
- 全额付款。
- 预付款。
- 采购退货抵扣。
- 到期提醒。
- 月度对账单。

余额必须由流水汇总，禁止直接修改余额。

## 4.9 日结

日结至少包含：

- 销售额。
- 现金实收。
- 微信实收。
- 支付宝实收。
- 银行卡实收。
- 赊账销售。
- 历史欠款回款。
- 退款。
- 手续费。
- 采购付款。
- 供应商还款。
- 报损成本。
- 预计毛利。
- 现金净流入。
- 系统与实际收款差异。

规则：

- 日结操作幂等。
- 确认后锁定。
- 禁止直接改历史日结。
- 只能补记、冲正、说明差异或由授权人员重新打开。

## 4.10 AI采购预测

输入因素：

- 历史销量。
- 星期和时段。
- 天气、温度、降雨。
- 节假日和市场活动。
- 当前库存。
- 批次剩余寿命。
- 在途采购。
- 供应商价格和起订量。
- 到货周期。
- 现金余额。
- 冷藏空间。
- 最大可承受损耗。
- 饭店和学校预订单。

输出必须包含区间、推荐值、成本、收入、损耗、毛利、缺货概率、积压概率和置信度。

## 4.11 AI建议可执行化

每条建议必须包含：

- 建议内容。
- 数据依据。
- 预计收益。
- 风险。
- 可信度。
- 可执行动作。
- 执行记录。
- 执行结果复盘。

可执行动作包括：

- 修改售价。
- 生成采购单。
- 生成清货任务。
- 锁定批次。
- 创建提醒。
- 创建供应商询价。

## 5. 技术实施规范

## 5.1 多租户规范

- 业务表必须包含 `merchant_id`，多门店业务还应包含 `store_id`。
- 禁止信任客户端直接传入的商户ID。
- 商户身份必须由JWT解析。
- 所有查询、更新、删除、聚合都必须带商户条件。
- 唯一约束优先采用 `(merchant_id, business_key)`。
- 跨商户隔离必须有自动化测试。

## 5.2 金额和数量规范

- 后端金额使用 `Decimal` 和数据库 `NUMERIC`。
- 禁止用float累计金额。
- 金额计算必须明确舍入规则。
- 数量支持小数，按品类配置精度。
- API可序列化为字符串或数值，但内部计算必须保持Decimal。
- 单位换算必须保存换算来源和本次实际换算值。

## 5.3 流水和冲正规范

- 库存、支付、应收、应付都采用追加流水。
- 已确认流水禁止物理删除。
- 修改已确认记录时生成冲正和新记录。
- 每条流水必须保存来源类型、来源ID、操作人、时间和幂等键。
- 余额和库存通过流水聚合或可靠投影得到。

## 5.4 幂等规范

- 所有可能被网络重试的写接口必须支持幂等。
- 幂等唯一范围默认为 `(merchant_id, idempotency_key)`。
- 支付回调还应按支付平台交易号唯一。
- 相同幂等键和相同请求返回原结果。
- 相同幂等键但请求内容不同返回409冲突。
- 幂等检查与业务写入必须处于同一事务。

## 5.5 事务和并发规范

- 创建订单、扣库存、写支付和写应收必须在同一事务完成。
- 库存不足必须整单回滚。
- 批量同步采用外层事务加每条保存点，单条失败不能破坏其他成功项。
- 并发扣库存使用行锁、原子条件更新或等价机制。
- 不允许捕获异常后无条件提交半成品数据。

## 5.6 API规范

统一响应示例：

```json
{
  "code": 0,
  "message": "",
  "data": {},
  "request_id": "optional"
}
```

规则：

- Pydantic强类型请求和响应。
- 禁止核心接口长期使用裸 `dict`。
- 参数错误返回422。
- 未登录返回401。
- 无权限返回403。
- 资源不存在返回404。
- 业务冲突返回409。
- 不可处理业务规则返回422或明确业务码。
- API错误不得泄露密钥、SQL和内部堆栈。

## 5.7 状态机规范

- 采购、销售、退款、批次、日结必须定义显式状态机。
- 禁止任意状态跳转。
- 每个状态变更记录操作人、时间和原因。
- 状态转换需要集中在服务层，而不是散落在路由中。

## 5.8 离线同步规范

状态：

```text
pending → syncing → synced
             ↘ failed → manual_retry → pending
```

规则：

- 本地先写入再更新UI。
- 客户端生成UUID和幂等键。
- 成功或服务端确认重复后才删除本地任务。
- 业务失败必须保留原因。
- 网络失败指数退避。
- 达到重试次数进入死信。
- 支持手动重试。
- 冲突不能简单覆盖，应通过流水合并或人工处理。

## 5.9 离线媒体规范

- 录音和拍照后使用 `wx.saveFile` 持久化。
- 服务端确认前不得删除本地文件。
- 保存任务类型、本地路径、业务载荷、幂等键、重试次数和错误原因。
- 上传时携带JWT。
- 上传成功后清理文件。
- 文件不存在时进入失败状态并提示重新采集。
- 限制本地总空间并提供可视化清理入口。

## 5.10 文件和对象存储规范

- 生产环境不把音频和图片永久保存在容器本地磁盘。
- 使用对象存储和受控访问URL。
- 校验MIME、扩展名、大小和内容类型。
- 文件名使用不可预测UUID。
- 凭证类文件记录上传人、商户、关联业务和保留期限。
- 禁止把对象存储密钥返回给小程序。

## 5.11 AI规范

- AI输出必须结构化并通过Schema校验。
- AI不得直接绕过业务服务修改数据库。
- AI动作调用与人工操作相同的服务层和权限校验。
- 保存输入数据版本、模型或规则版本、置信度和依据。
- 高风险动作必须人工确认。
- 允许忽略、撤销和复盘。
- 无真实数据时必须明确标记演示或低置信度，禁止伪装成真实预测。

## 5.12 硬件规范

- 设备拥有唯一ID并绑定商户和门店。
- 设备消息含序列号、时间戳和幂等键。
- 服务端处理重复、乱序、延迟和离线消息。
- 重量必须经过稳定窗口确认。
- 所有自动识别提供人工候选和手工兜底。
- 模拟设备和真实设备配置必须隔离。

## 5.13 数据库迁移规范

- ORM变化必须配套Alembic迁移。
- 迁移必须支持空库升级。
- 重要迁移必须验证升级、降级、再升级。
- 生产迁移避免一次性锁表过久。
- 新非空字段必须考虑旧数据回填。
- 不允许只改模型而不写迁移。

## 5.14 日志、审计和监控规范

- 每个请求生成request ID。
- 日志不得记录Token、密码、Secret和完整支付敏感数据。
- 关键业务写入审计日志。
- 监控登录失败、支付回调失败、库存不足、同步死信、日结差异和设备离线。
- 生产环境配置错误告警和健康检查。

## 5.15 小程序交互规范

- 核心操作优先控制在3步以内。
- 大按钮、明确金额和状态。
- 所有异步操作显示加载状态，防止重复点击。
- 明确展示待同步、同步中、同步失败。
- 失败不能只弹Toast，应提供可恢复入口。
- 不把技术错误直接展示给摊主，应转换为可理解的业务提示。
