## A. G1 多商户鉴权 · 技术方案（Product Brief for Eng）

> 配套 v0.1 §2 的 PRD 叙述，本文给工程可直接落地的接口、模型与改造清单。参考蓝本：`千摊智脑-auth参考实现.py`（REFERENCE ONLY，合入前需 code review）。

### A.1 认证链路
```
小程序 wx.login() ──> code
  ──POST /api/v1/auth/login {code}──> 后端 code2Session(微信) 换 openid
       ├─ openid 已绑 Merchant → 返回 JWT
       └─ 未绑 → 新建 Merchant(openid) → 返回 JWT + is_new=true
  JWT = HS256{ merchant_id, exp, type }，access 2h / refresh 7d
后续请求 Header: Authorization: Bearer <access_token>
```

### A.2 接口定义
| 方法 | 路径 | 入参 | 出参 | 说明 |
|---|---|---|---|---|
| POST | `/auth/login` | `{code: str}` | `{access_token, refresh_token, merchant_id, is_new}` | 微信 code 换 token；无 auth |
| POST | `/auth/refresh` | `{refresh_token: str}` | `{access_token}` | 续期；refresh 一次性（用后作废+签发新） |
| POST | `/auth/logout` | `Authorization` | `204` | 吊销 refresh（写黑名单/删记录） |
| GET | `/auth/me` | `Authorization` | `{merchant_id, created_at, consent_version}` | 当前身份自省 |

### A.3 数据模型变更
```sql
-- Merchant 扩展（Alembic 002 之后新增迁移）
ALTER TABLE merchants ADD COLUMN openid VARCHAR(64) UNIQUE;        -- 微信唯一标识
ALTER TABLE merchants ADD COLUMN created_at TIMESTAMP DEFAULT now();

-- 刷新令牌表（支持吊销/多设备）
CREATE TABLE auth_refresh_tokens (
  id UUID PK,
  merchant_id UUID FK,
  token_hash VARCHAR(128) UNIQUE,   -- 只存哈希，防泄露
  device_hint VARCHAR(32),
  expires_at TIMESTAMP,
  revoked_at TIMESTAMP NULL
);

-- 隐私同意记录（N2）
CREATE TABLE consent_records (
  id UUID PK,
  merchant_id UUID FK,
  version VARCHAR(16),              -- 如 "2026.1"
  agreed_at TIMESTAMP,
  withdrawn_at TIMESTAMP NULL       -- 撤回即填，触发被遗忘权流程
);
```

### A.4 改造清单（路由层身份可信化）
将下列接口从"信任 body/query 的 `merchant_id`"改为 `Depends(get_current_merchant)`：
- `routers/advice.py`：`get_daily_advice`、`simulate_scenario`（参考实现已示范）
- `routers/voice.py`、`routers/inventory.py`、`routers/vision.py`
- `routers/reports.py`、`routers/purchase.py`、`routers/stocktake.py`
- `routers/cloud.py`（经验云聚合，需保留差分隐私）
- **删除危险默认**：任何 `merchant_id = body.get("merchant_id", uuid.uuid4())` 一律移除。

### A.5 安全要点（Non-goals 反向约束）
- `jwt_secret` 仅存服务端配置，**绝不**下发前端；前端只持有 token。
- refresh token 哈希存储、一次性使用；登出即吊销。
- 全链路 HTTPS；`HTTPBearer(auto_error=False)` + 显式 401。
- **不**做"客户端可指定身份"的任何回退兼容；存量测试改用测试用 token 注入。

## B. 合规底线 PRD · N1 数据所有权/被遗忘权 + N2 隐私同意（P0/P1）

### B.1 N2 隐私同意（P0，随 G1 落地）
- **Why**：语音/图像/经营数据实时采集，使用前须明确告知并获授权，否则违反个保法第 13/17 条（告知—同意）。
- **用户故事**：作为新摊主，我希望首次启动时清楚看到"采集什么、用在哪、能否撤回"，以便自主决定是否授权。
- **需求**：
  - P0 首次启动 Consent 页（版本化文案），同意后写 `consent_records`。
  - P0 同意前拦截语音/图像采集入口，仅允许纯本地记账演示。
  - P1 设置页可查看同意状态 + 一键撤回（撤回触发 N1 删除流程）。
- **验收**：未同意 → 采集被拦；同意后 `consent_records` 有记录；撤回后采集再被拦。

### B.2 N1 数据所有权与可携/被遗忘权（P1）
- **Why**：摊主换机/退租/转行时数据无法带走或清除，违反个保法第 15 条（撤回同意）、第 45 条（删除权）、第 45 条便携权。
- **用户故事**：作为打算退出的摊主，我希望能一键导出全部经营数据备份，或彻底删除我的账号与数据，以便"我的数据我做主"。
- **需求**：
  - P1 `POST /account/export` → 异步打包 JSON/CSV（语音日志、库存、报表、建议），脱敏 openid。
  - P1 `POST /account/delete` → 软删除（`withdrawn_at` 触发）→ 定时任务匿名化/物理删除关联记录。
  - P1 删除前需二次确认 + 冷静期（7 天可撤销）。
- **Non-goals**：不导出其他商户数据；不提供他人数据批量导出；删除不波及聚合统计所需的差分隐私噪声桶（已匿名）。
- **验收**：导出包含该商户全量数据且无他人数据；删除后其 PII 不可恢复（审计日志保留脱敏副本）。

## C. G8 / G9 / G10 完整独立 PRD

### C.1 PRD-G8 · AI 经营问答（P3 · 复杂度高，独立立项）
- **Why**：摊主想要"小智帮我看明天该进多少西红柿"这类自然语言交互，而非在多个页间翻找。
- **目标用户**：习惯语音/对话的摊主。
- **核心设计**：
  - 受控意图识别：仅限经营域（记账/查询/建议/预测/对比/解释）。
  - 工具调用：识别意图后调 `/advice`、`/reports`、`/twin`、`/inventory`、`/cashier` 等已有接口，不重新实现逻辑。
  - 安全边界：经营域外问题一律"我只帮你看经营"式拒答；金额/操作类需二次确认才执行。
  - 上下文：单轮会话内保留最近 N 轮，支持"那西红柿呢？"指代。
- **需求优先级**：P3 意图路由 > 工具编排 > 多轮上下文 > 语音输入直连。
- **Non-goals**：不做通用聊天机器人；不答法律/医疗/经营外咨询；不自动执行不可逆操作（如删账）无确认。
- **验收**：经营域内问题正确调用工具并返回；域外问题被拒；执行类操作前弹确认；意图识别准确率（内部评测集）≥ 90%。
- **依赖**：G1 鉴权（身份）、现有各 router 接口稳定、评测集建设。
- **风险**：LLM 幻觉 → 工具返回值必须可溯源；成本 → 意图识别用小模型前置过滤。

### C.2 PRD-G9 · 预测评估看板（P2）
- **Why**：论文/竞赛答辩需"预测有效性"证据；商户侧也需信任预测才愿意采纳。
- **目标用户**：商户（信任建立）+ 竞赛/论文场景（证据展示）。
- **需求**：
  - P2 预测 vs 实际偏差看板：按日/品类展示 `predicted_qty` vs 实际销量，计算 MAE / MAPE。
  - P2 分层准确率：按品类、天气（晴/雨/高温）、节假日维度拆分的预测误差。
  - P2 误差分布与置信区间可视化（复用 `forecast.py` 的 `lower/upper_bound`）。
- **Non-goals**：不重新训练模型；不做实时预测（只看评估）。
- **验收**：看板展示历史预测与实际对比；MAE/MAPE 数值正确；分层切换无误。
- **依赖**：`services/forecast.py` 已支持分级预测并输出置信区间。

### C.3 PRD-G10 · Web 经营后台（P2）
- **Why**：高频复盘、对账、报税需大屏与导出；当前能力全锁在微信小程序，会计无法在电脑处理。
- **目标用户**：摊主（电脑复盘）、代账会计（导出 CSV）、竞赛演示（大屏）。
- **需求**：
  - P2 登录复用 G1 鉴权（Web 端 JWT/session）。
  - P2 报表 CSV 导出（日报/周报/趋势/排行）。
  - P2 大屏模式（核心指标 + 趋势 + 健康评分，竞赛可投屏）。
  - P3 对账视图（收支汇总、异常标记）。
- **Non-goals**：不重复小程序全部交互；不开放语音/图像采集（Web 非主场景）。
- **验收**：Web 登录后看到与小程序一致的报表；CSV 导出字段完整可打开；大屏布局适配 1920×1080。
- **依赖**：G1 鉴权；`reports.py` 接口已就绪。
- **风险**：Web 端需独立前端工程，排期另立。

## D. 更多新需求挖掘（N8+ · 以"商户数据私密"为透镜二次补盲）

> 在 v0.1 的 N1–N7 之外，继续从"个体赋能而非横向曝光"的角度挖。凡涉及社交/曝光的需求，均标 ⚠️ 需隐私审查。

| 编号 | 新需求 | Why（用户真实诉求） | 优先级 | 隐私标注 |
|---|---|---|---|---|
| N8 | **经营目标与自我里程碑** | 替代被砍的 G3 社交榜——摊主想"这月比上月多赚 20%"的自我达成，不涉及他人。 | P1 | ✅ 纯个体，隐私友好 |
| N9 | **多门店/连锁聚合** | 一个老板管多个摊点，需数据聚合看板但各店隔离。 | P2 | ⚠️ 同主体多身份，需明确归属边界 |
| N10 | **应收应付账本** | 已有供应商表但未建账本；赊账/欠款无记录，资金看板失真。 | P2 | ✅ 个体财务 |
| N11 | **智能补货与供应商直连** | 已有供应商表，采购清单可一键发给供应商/自动比价。 | P2 | ⚠️ 供应商信息共享需商户授权 |
| N12 | **现金流健康预警** | 强化 N4 风控：基于采购/销售/现金失衡提前预警"快没钱了"。 | P2 | ✅ 个体 |
| N13 | **损耗根因分析** | 减少浪费：识别"哪些品总损耗高 + 为什么（天气/备货过多）"。 | P2 | ✅ 个体 |
| N14 | **新手引导 / Onboarding** | 摊主上手门槛高，首次使用无流程化引导，流失在第一步。 | P1 | ✅ |
| N15 | **帮助中心 / 求助** | 遇到问题（识别错、记账乱）无自助求助入口，只能弃用。 | P1 | ✅ |
| N16 | **方言 / 自然语音记账** | 摊主方言重，当前 ASR 方言支持有限；"说一句就记一笔"体验缺口。 | P2 | ✅ 本地处理优先 |
| N17 | **营销工具（会员/优惠券）** | 复购提升：老客识别 + 满减/第二件半价模板。 | P3 | ⚠️ 涉及顾客信息，需最小化采集 |
| N18 | **电子发票 / 税务对接** | 报税痛点：经营数据可自动生成进销存台账供报税。 | P3 | ✅ 个体合规提效 |
| N19 | **设备迁移引导** | 换手机=丢数据焦虑（G2 教训延续）；需"一键迁移"体验。 | P1 | ✅ 复用 N6 备份 |
| N20 | **暗黑模式 / 视觉个性化** | 夜市摊主夜间使用刺眼；时段皮肤已有，缺全局暗色。 | P3 | ✅ |

**二次补盲结论**：
- **优先做（P1，体验+留存）**：N8 自我里程碑（直接填补 G3 砍掉后的"目标感"真空）、N14 新手引导、N15 帮助中心、N19 设备迁移。
- **P2 能力补强**：N9/N10/N12/N13 构成"资金+多店+供应链"的经营纵深；N11/N16/N17 为智能化延伸。
- **⚠️ 需隐私审查后再排**：N9（跨店边界）、N11（供应商共享）、N17（顾客信息）、N18（涉税但个体）— 上线前走隐私合规评审。
